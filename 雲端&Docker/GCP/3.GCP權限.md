# 3. GCP 身份與存取權限管理 (IAM)

> [!ABSTRACT]
> 本章介紹 Google Cloud Platform (GCP) 的 IAM (Identity and Access Management) 權限控制機制，解析基本角色、預定義角色與自訂角色的分類，探討服務帳戶（Service Account）的安全管理，並提供在 GCP 主控台內授予與修改權限的具體步驟。

---

## 一、 GCP IAM 核心概念

GCP 的 IAM 機制遵循最小特權原則（Principle of Least Privilege），其核心邏輯可以簡化為一個三元組關係：

$$\text{誰 (Who)} \rightarrow \text{能做什麼 (What)} \rightarrow \text{對哪項資源 (Which Resource)}$$

- **成員 (Member / Who)**：可以是個人 Google 帳號、群組，或是運行程式碼的服務帳戶。
- **角色 (Role / What)**：一組權限的集合（例如讀取、寫入或刪除）。
- **資源 (Resource / Which)**：如 VM 主機、Cloud Storage 儲存桶或資料庫。

---

## 二、 成員種類 (Who)

在 GCP 中，可以被授予權限的成員類型主要有以下幾種：

| 成員類型                                  | 說明                                    | 適用場景                            |
| :------------------------------------ | :------------------------------------ | :------------------------------ |
| **Google 帳號**                         | 綁定特定個人的 Google 信箱（如 `user@gmail.com`） | 開發人員、系統管理員個人登入                  |
| **服務帳戶 (Service Account)**            | 專門給「程式、VM 虛擬機或服務」使用的帳號，以金鑰進行驗證        | 伺服器程式調用 GCP API（如自動備份至 Storage） |
| **Google 群組**                         | 將多個 Google 帳號打包在一起的群組信箱               | 團隊管理（一次性對整個部門授予權限）              |
| **Google Workspace / Cloud Identity** | 企業組織所屬的專屬域名（如 `employee@company.com`） | 企業內部員工統一身分驗證與權限控管               |

---

## 三、 角色分類與權限層級 (What)

GCP 提供了三種不同精細度的角色類型，以滿足不同的管理需求：

### 1. 基本角色 (Primitive Roles)
這是最早期、粗粒度的權限設定，直接影響專案內的所有資源：
- **Viewer (檢視者)**：唯讀存取權。只能查看資源，不能做任何修改。
- **Editor (編輯者)**：修改與編輯權。包含 Viewer 權限加修改資源，但不能更動 IAM 權限或帳務。
- **Owner (擁有者)**：完全控制權。包含 Editor 權限加管理權限（IAM 授權）與專案帳務管理（Billing）。

> [!WARNING]
> **安全性警示**
> 基本角色的權限範圍過大（例如 Editor 角色可以刪除專案內所有的虛擬主機與資料庫）。在生產環境中，**強烈反對**給一般成員授予基本角色，應改用預定義角色。

### 2. 預定義角色 (Predefined Roles)
GCP 針對各個不同的雲端服務，量身打造的精細角色。這能讓您只授予成員特定服務的存取權：
- 例如：`Compute Instance Admin` (可完整管理虛擬機，但無法碰觸資料庫)。
- 例如：`Storage Object Viewer` (只能讀取 Cloud Storage 中的檔案，無法上傳或刪除)。

### 3. 自訂角色 (Custom Roles)
當預定義角色仍無法滿足特殊安全需求時，企業可以手動挑選最底層的 API 權限，組合出客製化的角色。
* *限制*：需要人工維護 API 權限的更新，且無法應用於組織級別以外的部分第三方資源。

## 四、 如何在主控台 (Console) 設定權限

以下是為新的團隊成員或協作夥伴授予專案權限的具體步驟：

### 步驟 1：進入 IAM 介面
1. 登入 [GCP 主控台](https://console.cloud.google.com/)。
2. 點選左上角主選單 ➔ **「IAM 和管理 (IAM & Admin)」** ➔ 點選 **「IAM」**。
3. 確保右上方下拉選單已切換至正確的**專案 (Project)**。

### 步驟 2：授予存取權
1. 點擊頁面頂部的 **「授予存取權 (GRANT ACCESS)」** 按鈕。
2. 在 **「新增主體 (New principals)」** 欄位中，輸入被授權者的 Email 或服務帳戶名稱。
3. 在 **「選取角色 (Select a role)」** 下拉選單中，依據需求選擇對應的服務與精細角色（例如 `Compute Engine` ➔ `Compute Viewer`）。
4. （選用）可點擊「新增其他角色」以一次賦予多個權限。
5. 點擊 **「儲存 (Save)」**，權限將在數秒內生效。

## 五、 服務帳戶 (Service Account) 的建立與應用

服務帳戶是程式之間進行安全通訊的重要橋樑。

### 1. 建立服務帳戶
1. 在左側選單點選 **「服務帳戶 (Service Accounts)」** ➔ 點擊頂部 **「建立服務帳戶」**。
2. 輸入自訂的服務帳戶名稱與描述，系統會自動生成對應的 Email。
3. 點選「建立並繼續」，為此服務帳戶選擇角色（例如：`Storage Object Admin`）。
4. 點選「完成」。

### 2. 生成並下載金鑰金鑰 (JSON Key)
1. 點擊剛建立的服務帳戶進入詳情頁。
2. 切換至 **「金鑰 (Keys)」** 分頁 ➔ 點擊 **「新增金鑰」** ➔ 選擇 **「建立新金鑰」**。
3. 選擇 **`JSON`** 格式，點擊建立。
4. 瀏覽器會自動下載一個 JSON 格式的金鑰檔案。

> [!CAUTION]
> **私鑰外洩風險**
> 下載的 JSON 金鑰代表此帳戶的最高權限，**絕對不能上傳至 GitHub 或公開代碼庫**。如果金鑰外洩，必須立刻在 GCP 控制台點擊「刪除」該金鑰以防止未授權存取。

### 3. 程式碼調用驗證實務
在程式中使用環境變數指向該金鑰路徑，GCP SDK 會自動載入權限進行驗證：

```bash
# 在 Linux/macOS 伺服器終端機中設定環境變數
export GOOGLE_APPLICATION_CREDENTIALS="/path/to/your/service-account-key.json"
```

```python
# Python 程式碼直接調用 SDK (無須在代碼中寫死金鑰路徑)
from google.cloud import storage

def list_buckets():
    # 系統會自動讀取 GOOGLE_APPLICATION_CREDENTIALS 環境變數進行驗證
    storage_client = storage.Client()
    buckets = list(storage_client.list_buckets())
    for bucket in buckets:
        print(bucket.name)
```

## 六、 GCP Agent Platform 權限控管 (Dialogflow & Vertex AI Agent Builder)

在 GCP 中開發與部署對話式 AI、Chatbot（如 LINE Bot）或生成式 AI 搜尋 Agent 時，主要會涉及兩個核心的 Agent 平台：**Dialogflow** 與 **Vertex AI Agent Builder**。

這兩個平台都擁有專屬的 IAM 預定義角色，以利進行精細的存取控制。

### 1. Dialogflow Agent 平台角色
當您的 Web 服務、LINE Bot 或客服系統需要與 Dialogflow Agent 通訊時，必須為對應的服務帳戶授予以下角色之一：

- **Dialogflow API Admin (`roles/dialogflow.admin`)**：
  - **權限範圍**：最高管理限。可建立、編輯、訓練、刪除 Agent，以及修改所有意圖（Intents）與實體（Entities）。
  - **適用場景**：開發與測試階段的管理人員。
- **Dialogflow API Client (`roles/dialogflow.client`)**：
  - **權限範圍**：可以向 Agent 發送偵測意圖（Detect Intent）請求與其進行問答互動，但無法修改 Agent 的任何配置。
  - **適用場景**：**LINE Bot 或網頁前端串接最常用的生產環境角色**（基於最小特權原則）。
- **Dialogflow API Reader (`roles/dialogflow.reader`)**：
  - **權限範圍**：只能唯讀查看 Agent 的配置與設定。

### 2. Vertex AI Agent Builder (GenAI Agent) 角色
用於建置企業級 RAG (檢索增強生成) 搜尋與問答 Agent：

- **Discovery Engine Admin (`roles/discoveryengine.admin`)**：
  - **權限範圍**：具備 Agent Builder 的完整管理權，包含資料庫綁定、自訂代碼布建與模型微調。
- **Discovery Engine Editor (`roles/discoveryengine.editor`)**：
  - **權限範圍**：可編輯應用與資料庫配置。
- **Discovery Engine Viewer (`roles/discoveryengine.viewer`)**：
  - **權限範圍**：僅能進行搜尋測試與查看設定。
- **Agent Platform User (`roles/agentplatform.user`)**：
  - **權限範圍**：提供在 Gemini Enterprise Agent Platform 中使用、管理與部署 AI Agent 資源的權限。
  - **適用場景**：專門用於 Agent Platform 的開發、測試與運行。
- **Vertex AI User (AI Platform User - `roles/aiplatform.user`)**：
  - **權限範圍**：提供調用與運行 Vertex AI 資源（如 Gemini LLM 模型 API）的權限，以利測試與訓練。
  - **適用場景**：**開發人員帳戶或 IDE 智慧插件（如 Antigravity）串接 Gemini / Vertex AI 時所需的預設必備授權角色**。

### 3. Service Agent (服務代理人) 的自動授權
GCP Agent Platform 在讀取您的資料來源（如掛載於 Cloud Storage 的 PDF 文件庫）時，系統會自動在 IAM 中新增一個名為 **「Google Cloud Service Agent」** 的特殊系統帳戶，並為其分配對應的服務代理人角色。
- *注意事項*：切勿手動刪除該服務代理人的存取權，否則會導致 Agent Platform 無法正常抓取資料庫或儲存桶中的 RAG 語料。
