# 6. 優化原理與神經網路驗證

> [!ABSTRACT]
> 本章介紹機器學習的優化原理，包括泛化能力、過擬合與欠擬合的識別與解決方法；並詳細說明如何使用 Keras API 進行模型編譯與訓練（包含 `compile()`、`fit()`、`predict()`、`evaluate()`），以及透過 `EarlyStopping` 與 `ModelCheckpoint` 等 Callbacks 來調控訓練過程與儲存網路模型。

---

## 一、 優化原理

### 1. 泛化能力
泛化能力 (Generalization Ability) 是指一個機器學習算法對於**沒有見過的樣本資料**具有良好識別能力。也就是此算法在定義的範圍內都能夠實用。
- 對於機器學習或深度學習的模型而言，我們不僅要求它對訓練數據集有較小的訓練誤差，同時也希望它可以對未知數據（測試集）也有很好的擬合結果（泛化能力）。
- 模型在未知數據上的誤差被稱為**泛化誤差**。

### 2. 過擬合 (Overfitting)
當模型過度地學習訓練樣本中的細節與噪音，把訓練樣本自身的一些特點當作了所有潛在樣本都會具有的一般性質，就會導致泛化效能的下降，以至於模型在新的資料上表現很差。

#### 如何識別模型過擬合
- 當給定一個訓練數據集，如果模型的複雜度過低，很容易出現欠擬合狀況；如果模型複雜度過高，很容易出現過擬合狀況。因此解決欠擬合和過擬合的一個辦法是針對數據集選擇合適複雜度的模型。
- 以訓練集跟驗證集的訓練週數與準確度來說（如下圖）：
  - 模型在訓練過程中對於訓練資料反覆學習，因此訓練集的準確度會逐漸上升。
  - 驗證資料集不會參與訓練，因此過了某個週期後（如圖中中間那條虛線），驗證集的準確度會開始下降。這個狀況代表模型開始產生過擬合，此時可以提早停止訓練以避免過擬合。

![[{1C84E626-CC65-41F9-9BA6-4BC9A2C15F18}.png|283]]

### 3. 欠擬合 (Underfitting)
指模型對訓練樣本的一般性質尚未學好，導致在訓練集和驗證集上的表現都不理想。

## 二、 使用 Keras API 撰寫網路

建構好模型後，接下來必須用 `compile()` 函數指定優化器、損失函數及成效衡量指標。

### 1. 指定優化函數 (Optimizer)
```python
model.compile(optimizer=優化器, loss=損失函數, metrics=["準確度"])  # 方法一
```
- **指定方式**：
  - **字串形式** (方法一)：給出優化器的名稱。
  - **函數形式** (方法二/三)：使用 Keras 優化器類別，此方式可以進一步設置學習率、動量等超參數。

![[{E8C3B96C-1A3C-49E3-98E4-08C06E0B3E4A}.png|619]]
![[Pasted image 20260611141824.png]]

### 2. 選擇監測性能指標 (Metrics)
`metrics` 參數用來在訓練過程中監測性能指標（評價模型好壞的指標），指定的方式有兩種：

![[{3343442D-955A-4479-9919-12BAE6FE2EB6}.png|483]]

- **指標種類說明**：
  - **Categorical Accuracy**：要求樣本在真值類別上的預測分數是所有類別預測分數的最大值，才算預測正確。傳統的準確率（Top-1）是看模型預測機率最高的「第一名」答案是否正確。
  - **Top-K Categorical Accuracy**：放寬了標準，只要正確答案落在模型預測機率最高的「前 K 名」之內，即算模型猜對。

### 3. 模型的訓練：fit() 函數
當模型編譯完成之後，接下來就要使用 `fit()` 函式進行訓練動作。 
- `Keras Model.fit()` 方法會傳回一個 `History` 對象。
- `history.history` 屬性是一個記錄了連續迭代的訓練/驗證（如果存在）損失值和評估成效的字典。

### 4. 預測函式 predict()
用於對新輸入的數據進行預測，獲取模型的輸出值。

### 5. 評估函式 evaluate()
計算模型在測試集或驗證集上的損失值與指標。

> [!NOTE]
> **載入模型後的 evaluate() 注意事項**
> 如果載入模型後想用評估函數 `evaluate()`，則必須在載入模型後重新編譯（`compile()`）。
> 因為 `predict()` 不需要評估任何指標或損失，它只是將輸入數據傳遞過模型並獲取輸出；但 `evaluate()` 會計算損失函數和指標，這些是在 `compile()` 中設定的。

## 三、 訓練控制與 Callbacks 機制

### 1. Early Stopping (提前停止)
在進行監督式學習的過程中，這是一種尋找模型收斂時機點並避免過擬合的有效方法。

在 `tf.keras` 中，提供了下列參數可設置：
```python
tf.keras.callbacks.EarlyStopping(
    monitor='val_loss', 
    min_delta=0, 
    patience=0, 
    verbose=0, 
    mode='auto'
)
```

1. **`monitor`**：設定監控的數據。除了 `loss` 外，其他可監控的數據與 `metrics` 設定的指標相關（例如 `'acc'`, `'val_acc'`, `'loss'`, `'val_loss'` 等）。
   > [!TIP]
   > 正常情況下若有驗證集，建議使用 `'val_acc'` 或 `'val_loss'`。
2. **`min_delta`**：評斷監控數據是否有改善的閾值。只有當數據變動幅度大於 `min_delta` 才算是有改善。
3. **`patience`**：容忍監控數據在多少個 epoch 內沒有出現改善。
   - `patience` 的設置與 `min_delta` 相關。一般來說，`min_delta` 設得小時，`patience` 可以相對調低；反之，若 `min_delta` 大，則 `patience` 需加大。
   - **注意**：若 `patience` 設得太小，可能導致模型在訓練前期（還在全域搜尋時）就被迫停止；若設得太大，則失去 EarlyStopping 的意義。
4. **`verbose`**：詳細程度，可設定為 `0` 或 `1`。若設定為 `1`，則會在 EarlyStopping 發生後，以文字顯示在哪個訓練週期發生（例如：`"Epoch 00003: early stopping"`）。
5. **`mode`**：有 `auto`、`min` 和 `max` 三種設置選擇。用來設定監控數據的改善方向：
   - 若監控數據是越大越好（如 `acc`），則設置為 `max`。
   - 若監控數據是越小越好（如 `loss`），則設定為 `min`。
   - `auto` 模式會自動從被監控數據的名稱中判斷方向。

## 四、 網路的儲存與載入

Keras 提供了幾種保存模型的方式：

![[{A0EEA876-0E37-4B2E-AEAB-240C6F5D0481}.png|535]]

### 1. 方法一：保存完整網路模型與權重參數 (HDF5 格式)
可以使用 `model.save(filepath)` 將 Keras 模型和權重保存在一個 HDF5 文件中（副檔名通常為 `.h5`）。該文件包含：
- 模型的結構（以便重構該模型）
- 模型的權重
- 訓練配置（損失函數、優化器等）
- 優化器的狀態（以便於從上次訓練中斷的地方繼續訓練）

#### 讀取模型：
```python
tf.keras.models.load_model('model_name', compile=False)
```
- **重新編譯**：使用 `load_model` 函數加載模型時，如果需要對該模型進行**繼續訓練或評估**，需要將 `compile` 設為 `True`（此為預設值），或在加載後重新編譯。如果只是需要對模型進行**預測**，則不需要編譯模型，可將 `compile` 設為 `False`。

### 2. 方法二：SavedModel 格式 (只保留模型)
通過 `tf.saved_model.save(network, path)` 即可將模型以 SavedModel 方式保存到指定的目錄（資料夾）中：
```python
tf.saved_model.save(network, path)
```

### 3. 方法三：只保留權重參數
這種儲存與載入網路的方式最為輕量級，文件中保存的僅僅是張量參數的數值，並沒有其它額外的結構參數。
- **限制**：載入時需要使用相同的網路結構才能夠正確恢復網路狀態，因此一般在擁有網路源文件（代碼結構）的情況下使用。
```python
model.save_weights(filepath)
```

### 4. 繼續訓練模型 (Fine-tuning / Resume Training)
在實務上，新資料會不斷進入資料庫，模型也會慢慢老化，導致預測效果變差。此時我們必須載入舊的模型參數，以新數據加以訓練（更新模型），確保模型表現符合預期。

## 五、 ModelCheckpoint
此 Callback 的功能是在訓練的指定時刻（如每個 Epoch 結束時）自動將神經網路模型儲存起來。

```python
keras.callbacks.ModelCheckpoint(
    filepath, 
    monitor='val_loss', 
    verbose=0, 
    save_best_only=False, 
    save_weights_only=False, 
    mode='auto', 
    save_freq='epoch'
)
```

1. **`filepath`**：保存模型的路徑字串。
2. **`monitor`**：被監測的數據名稱。
3. **`verbose`**：詳細信息模式，`0` 或 `1`。
4. **`save_best_only`**：若設為 `True`，則只有在被監測數據達到最佳時才會覆蓋保存，避免最佳模型被後續表現較差的模型覆蓋。
5. **`mode`**：有 `auto`、`min`、`max` 三種選擇。配合 `save_best_only=True` 使用，決定被監測數據的判定方向（如 `val_acc` 為 `max`，`val_loss` 為 `min`）。
6. **`save_weights_only`**：若為 `True`，則只保存模型權重（相當於調用 `model.save_weights(filepath)`）；若為 `False`，則保存整個模型（相當於調用 `model.save(filepath)`）。
7. **`save_freq`**：預設為 `'epoch'`，代表在每個 Epoch 結束時儲存模型。
