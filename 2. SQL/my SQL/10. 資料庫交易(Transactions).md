# 資料庫交易與控制 (Database Transactions)

> [!ABSTRACT]
> 介紹資料庫交易的核心 ACID 特性，並學習在 MySQL 中如何使用 COMMIT、ROLLBACK 與儲存點進行交易控制。

---

交易 (Transaction) 是多個 DML 指令所構成的邏輯工作單位，主要用於維持資料庫中資料的一致性 (Consistency) 和正確性 (Correctness)。

---

## 1. 交易控制的核心概念

### 1.1 為什麼需要交易控制？
當我們對資料庫進行多筆資料修改時，必須確保這些操作是「一組不可分割的任務」：
- **全部成功**：交易期間所修改的所有資料正式寫入並永久保存於資料庫中。
- **全部失敗**：若執行期間發生任何錯誤，必須取消或回復 (Rollback) 該交易期間內所有的修改，使資料庫回到交易前的狀態。
- ==**All or Nothing**：交易的結果只有「全部成功」或「全部失敗」兩種情況，絕不存在中間狀態。==

### 1.2 交易的 ACID 特性
所有合格的關係型資料庫交易都必須嚴格符合 **ACID** 理論：

1. **原子性 (Atomicity，不可分割性)**
   - 交易必須被視為一個整體的最小工作單元。交易中的所有操作要麼全部完成，要麼全部不執行。若有任何一部分失敗，整筆交易皆會失效。
2. **一致性 (Consistency)**
   - 交易前後，資料庫必須從一個「一致性狀態」轉換到另一個「一致性狀態」，維持所有的完整性限制（例如餘額不能為負、外來鍵關聯必須存在）。
3. **隔離性 (Isolation，獨立性)**
   - 多個交易同時併發執行時，各交易之間不應互相干擾。每個交易都應該像是在獨立、專屬的環境中執行一樣。
4. **持久性 (Durability)**
   - 交易一旦成功提交 (Commit)，其對資料庫所做的變更就是永久性的，即使系統隨後崩潰，資料也不會遺失。

## 2. MySQL 的交易處理機制

### 2.1 引擎限制
> [!IMPORTANT]
> - 在 MySQL 中，只有 **InnoDB** 儲存引擎支援交易處理，早期的 **MyISAM** 引擎並不支援交易。

### 2.2 交易控制模式 (AUTOCOMMIT)
MySQL 預設會開啟自動確認模式：
- **自動確認 (AUTOCOMMIT = 1)**：
  - 系統變數 `@@AUTOCOMMIT` 預設為 `1` (True)。
  - 每執行完一個 DML 指令（如 `INSERT`、`UPDATE`、`DELETE`），系統會立刻將變更提交給資料庫永久保存。
  - **查詢指令**：`SELECT @@AUTOCOMMIT;`
- **手動確認 (AUTOCOMMIT = 0)**：
  - 可透過指令設定：`SET AUTOCOMMIT = 0;`
  - 在此狀態下，DML 指令執行後僅會寫入暫存區，除非執行 `COMMIT`，否則不會永久寫入。若執行 `ROLLBACK` 則可撤銷修改。

### 2.3 自訂交易區塊
不論 `AUTOCOMMIT` 的狀態為何，我們都可以使用 `START TRANSACTION` 顯式開啟一個交易區塊：

```sql
START TRANSACTION; -- 或 BEGIN;
-- DML 指令序列
dml_statement1;
dml_statement2;
-- 結束交易
COMMIT; -- 提交並保存變更
-- 或
ROLLBACK; -- 放棄變更
```
- **狀態恢復**：自訂交易結束後，交易控制模式將恢復到執行 `START TRANSACTION` 之前的狀態。

## 3. 資料在交易過程中的狀態

了解資料在交易不同階段的能見度與鎖定狀態，對於併發控制非常重要：

### 3.1 交易進行中 (Uncommitted State)
- **資料可見度**：僅有當前的連線 (Session) 可以看到修改後的資料，其他 Session 僅能讀取到修改前的舊值（隔離性表現）。
- **列鎖定機制 (Row Locking)**：
  - 交易中被修改的資料列會被加上「排他鎖 (Exclusive Lock)」。
  - 其他 Session 無法對這些被鎖定的資料列進行 `DML` 修改，但**不影響**一般的 `SELECT` 查詢。
- **不可包含 DDL**：交易進行中不可執行 DDL（如 `CREATE`、`ALTER`），否則系統會自動執行隱式提交 (Implicit Commit) 並結束交易。

### 3.2 確認交易 (COMMIT) 後
- 資料變更正式永久寫入硬體儲存。
- 釋放交易期間所產生的列鎖定 (Row Locks)。
- 刪除該交易中設定的所有儲存點 (Savepoints)。
- 其他 Session 立即可見修改後的最新資料。
- 交易正式結束，資料無法再進行 `ROLLBACK`。

### 3.3 放棄交易 (ROLLBACK) 後
- 取消交易中所有暫存的資料變更，還原回交易前的狀態。
- 釋放列鎖定 (Row Locks)。
- 清除所有儲存點。
- 其他 Session 可以對該資料進行操作。
- 交易正式結束。

### 3.4 交易儲存點 (SAVEPOINT)
當交易規模較大時，可使用儲存點進行「部分斷回」，而不需要放棄整筆交易。

```sql
SAVEPOINT point_name;           -- 設定儲存點
ROLLBACK TO SAVEPOINT point_name; -- 退回到指定的儲存點，此時交易「不會結束」
RELEASE SAVEPOINT point_name;   -- 移除儲存點
```
- ==**特性**：==
  - 儲存點名稱**區分大小寫**。
  - `ROLLBACK TO SAVEPOINT` 會將狀態退回到該儲存點，並遺棄該點之後的所有變更，但交易本身仍在進行中，仍需最後執行 `COMMIT` 或 `ROLLBACK` 來結束交易。

## 4. 併發交易衝突與隔離級別

多個 Session 同時操作相同資料時，可能會引發不同的衝突事件。

### 4.1 併發衝突事件
- **髒讀 (Dirty Read)**：
  - A 交易修改了資料但**尚未提交**，B 交易讀取了該筆修改。隨後 A 交易發生錯誤執行了 `ROLLBACK`。此時 B 交易讀取到的資料即為根本不存在的「髒資料」。
- **更新遺失 (Lost Update)**：
  - 兩個交易同時讀取同一數值，並各自在本地計算後更新回資料庫，導致其中一個交易的更新覆蓋了另一個交易的更新。
- **不可重複讀 (Non-Repeatable Read)**：
  - A 交易在同一個交易中讀取同一個欄位兩次，但在此期間 B 交易修改或刪除並提交了該資料，導致 A 交易兩次讀取出的結果不一致。
- **幻讀 (Phantom Read)**：
  - A 交易讀取某個範圍的資料列（如 `age > 20`），B 交易在此範圍內 `INSERT` 並提交了新資料。當 A 交易再次查詢該範圍時，發現多出了像「幽靈」一樣憑空出現的資料列。

### 4.2 交易隔離級別 (Isolation Levels)
為解決上述衝突，SQL 標準定義了四種隔離級別。安全級別越高，併發效能越低：

| 隔離級別 | 髒讀 (Dirty Read) | 不可重複讀 (Non-Repeatable Read) | 幻讀 (Phantom Read) | 鎖定機制與效能 |
| :--- | :---: | :---: | :---: | :--- |
| **Read Uncommitted**<br>(讀未提交) | ❌ 可能發生 | ❌ 可能發生 | ❌ 可能發生 | 效能最高，但完全沒有資料安全性。 |
| **Read Committed**<br>(讀已提交) | 安全 | ❌ 可能發生 | ❌ 可能發生 | 大多數資料庫（如 Oracle, SQL Server）的**預設級別**。 |
| **Repeatable Read**<br>(可重複讀) | 安全 | 安全 | ❌ 可能發生 | **MySQL (InnoDB) 的預設級別**。透過 MVCC 機制，MySQL 在此級別已大幅防範了幻讀。 |
| **Serializable**<br>(序列化 / 串行化) | 安全 | 安全 | 安全 | 效能最低。所有交易排隊執行，完全避免併發問題。 |
