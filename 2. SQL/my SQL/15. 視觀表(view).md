# 虛擬資料表與視觀表 (Views)

> [!ABSTRACT]
> 認識視觀表 (View) 的定義與優點，學習如何建立、使用與更新視觀表以簡化複雜查詢。

---

視觀表 (View) 是一個「虛擬的資料表」，它本身不儲存任何實體資料，而是由一個 SELECT 查詢敘述動態定義出來的邏輯視圖。被 View 所查詢的資料表稱為**基底資料表 (Base Tables)**。

- ==**核心概念**：View 僅儲存 SQL 查詢的「定義」，真正的資料仍安全地儲存在基底資料表中。==

---

## 1. 使用視觀表 (View) 的優勢與用途

1. **資訊隱藏與加強安全性**：
   - 可以將實體資料表的多個敏感欄位隱藏，限制使用者只能透過 View 存取特定的非敏感欄位。
   - 檢視表可以設定為**唯讀 (Read-only)**，防止外部使用者直接透過 DML 操作去更改底層的實體資料。
2. **簡化複雜查詢**：
   - 將包含多張表格 JOIN、複雜子查詢或統計計算的複雜 SELECT 語句包裝在 View 中，後續開發者僅需像查詢單一表格一樣調用該 View 即可。
3. **邏輯資料獨立性**：
   - 當底層的基底資料表結構有變更（如修改欄位名稱、拆分資料表）時，僅需要重新修改 View 的 SQL 定義，而無需修改任何與資料庫對接的外部應用程式碼。
4. **資料呈現多樣化**：
   - 針對同一個基底資料表，可以針對不同部門、角色或需求，客製化建立多個不同的 View。

---

## 2. 建立與替換視觀表 (CREATE VIEW)

### 2.1 基礎語法
```sql
CREATE [OR REPLACE] VIEW view_name [(column_alias_list)] 
AS 
select_statement
[WITH CHECK OPTION];
```
- **`OR REPLACE`**：如果同名的 View 已經存在，系統會自動覆蓋並取代它（等同於執行 `ALTER VIEW`）。
- **欄位別名清單 `column_alias_list`**：可以自訂 View 輸出的表頭欄位名稱。

### 2.2 語法限制
- **FROM 子句限制**：在建立 View 時，`FROM` 子句中**不可以使用子查詢 (Subquery)**。
- **ORDER BY 限制**：定義 View 時不要放入 `ORDER BY` 子句；即使寫入，在 View 被調用查詢時也會被系統自動忽略。

---

## 3. 視觀表種類與 DML 修改規範

視觀表依據定義時的複雜度，可分為簡單與複雜兩類，這直接影響了該 View 是否支援 DML 修改（如 `INSERT`、`UPDATE`、`DELETE`）：

### 3.1 簡單視觀表 (Simple Views)
- **特徵**：
  - 僅基於**單一**基底資料表進行查詢。
  - 沒有使用 `JOIN` 連結多張表。
  - 沒有使用群組聚合函數（如 `SUM()`、`AVG()`）或進行資料分組 (`GROUP BY`)。
- **DML 修改特性**：屬於**可更新的檢視表 (Updateable View)**，允許對其執行新增、修改與刪除，變更會直接同步回基底表。

### 3.2 複雜視觀表 (Complex Views)
- **特徵**：
  - 基於**一個以上**的基底資料表進行查詢 (使用了 `JOIN`)。
  - 使用了聚合函數、常值計算或 `GROUP BY` 等分組運算。
- **DML 修改特性**：**不可更新**。對其進行任何 DML 修改均會引發語法報錯。

### 3.3 查詢 View 是否可更新 (Is Updatable)
可以透過查詢 MySQL 系統字典表，確認特定 View 是否支援 DML 修改：

```sql
SELECT table_name, is_updatable
FROM information_schema.views;
```

---

## 4. DML 修改的細部限制與 CHECK OPTION

當對可更新的簡單視觀表執行變更時，需遵守以下規範：

### 4.1 欄位變更規則
- **非計算欄位**：可以直接修改。
- ==**計算欄位**：絕對不能修改。==
- **複合/複雜檢視表**：無法執行任何修改、新增與刪除。
- **刪除資料限制**：如果要刪除的資料在基底表中被其他資料表的外來鍵所參考，會因為違反關聯完整性而無法刪除。

### 4.2 WITH CHECK OPTION 選項
用來對 View 的 DML 變更進行邊界約束。
- **機制**：若加上此選項，在透過 View 執行 `INSERT` 或 `UPDATE` 時，**變更後的新資料必須符合 View 中 `WHERE` 子句所設定的篩選條件**，否則系統會拒絕寫入。

```sql
WHERE <condition>
WITH CHECK OPTION;
```

---

## 5. 刪除與修改視觀表

### 5.1 修改 View 定義 (ALTER VIEW)
若需要微調已建立的 View，可以使用 `ALTER VIEW`。這比刪除重建更好，因為它可以完整保留原本已賦予該 View 的所有使用者權限。

```sql
ALTER VIEW view_name [(column_list)] 
AS 
select_statement
[WITH CHECK OPTION];
```

### 5.2 刪除 View (DROP VIEW)
==**安全警示**：`DROP VIEW` 僅會將該虛擬表格的定義從資料庫中移除，絕對不會影響或刪除基底資料表中的任何實體資料。==

```sql
DROP VIEW view_name;
```
