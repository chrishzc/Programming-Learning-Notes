# 資料庫設計程序說明

> [!ABSTRACT]
> 學習資料庫設計的生命週期，掌握實體-關係模型 (ER Model) 的核心名詞（實體、屬性、關係）與設計程序。

---

1. 需求的搜集與分析: 與資料庫的使用者面談以了解並記錄其需求、搜集資料要詳細與完整
2. 概想資料庫設計 −以高階的、概想的資料模型來產生概想的資料庫(schema) ，將使用者需求以簡明、一致的方式表達 ，確定能符合使用者的模型例如：使用ER模型
3. 資料模型轉換 −選定某一種DBMS (關聯式，NoSQL) −將邏輯資料庫構造轉換為選定的DBMS所提供的資料庫構造
4. 實體資料庫設計 −選定特定資料庫系統(MySQL, SQL Server ..) −根據選定的DBMS，決定資料庫裡的儲存結構與檔案組織
## ER模型
概想資料模型，以儘量接近使用者的觀點來描述資料，而非著重在資料的存放方式
ER模型:實體—關係模型(Entity-Relationship model) 
- 是一種概想資料模型 −目前大都用在資料庫的設計過程中
- P. Chen在1976年提出
### 名詞譯義
1. 實體(Entity):用來描述實際存在的事物(例:員工)，也可以是邏輯抽象的概念(例:部門) −可以再細分成強實體及弱實體(依附強實體而存在，例:家屬)
2. 屬性(Attribute):用來描述實體的性質，每個屬性必須定義其值域
	   - 鍵值屬性:該屬性的值在某個環境下具有唯一性(例:員工編號)
	   - 合成屬性:該屬性可由其他更小的屬性組成(例:姓名可由姓及名所組成)
	   - 多值數性:該屬性可以有多個值(例:電話)
	   - 衍生屬性:該屬性可以由其他屬性推導出來(例: 年齡)
3. 關係(relationship): 用來表達兩個實體間的關聯性
	   - 關係還具有基數(Cardinality)，代表實體所能參與關係的實例數，通常分成三類: 一對一關係、一對多關係、多對多關係
	   - 弱實體與其依附實體間之關係特稱為確認關係
### 範例
Company資料庫的初步構造，先定義Company資料庫的實體，根據使用者的需求定義了四個實體
1. DEPARTMENT: 
   Name, Number, {Locations}, Manager, ManagerStartDate 
2. PROJECT: 
   Name, Number, Locations, ControllingDepartment 
3. EMPLOYEE: 
   Name(Fname, Lname), ENo, Address, Sex, Salary, Birthday, Department, Supervisor, {WorkOn(Project, Hours)} (多值屬性用{ }括起來、合成屬性用( )括起來)
4. DEPENDENT:
   Employee, DependentName, Sex, Birthdate, Relationship 

- 每位員工參與專案是以EMPLOYEE之WorkOn(Project, Hours)這個多值屬性來描述。 
- 儘量在定義實體時讓它的鍵值為基本屬性。
- 在上述的定義裡我們還要找出實體間的關係，像DEPARTMENT裡的 manager是EMPLOYEE裡的一員，PROJECT裡的ControllingDepartment是 DEPARTMENT裡的一員…..等。
- ==事實上，關係是屬性的一種特例==，我們可以在屬性中找到關係，本例中就有 六個關係。
